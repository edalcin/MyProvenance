# Backup/restauração do SQLite na área administrativa, sem reiniciar o processo

A área administrativa (ADM_PWD, docs/especificacao.md §9) precisava de backup/restauração do SQLite — mesmo padrão já usado em outros projetos do autor (Go: `notas`, onde a restauração troca o `*sql.DB` por baixo de um ponteiro compartilhado, sem reiniciar o processo). Portar esse padrão literalmente para este stack não funciona: `better-sqlite3` não tem pool nem API de reconexão, e cada repository (`registros.ts`, `agentes.ts`, `usuarios.ts` etc.) prepara `Statement`s **uma única vez, no import do módulo**, presos à conexão nativa original — fechar essa conexão e abrir outra deixaria todos os `Statement`s já preparados apontando para um handle morto, exigindo reiniciar o processo pra tudo voltar a funcionar (e depender de política de restart do container, que não é garantida em toda instalação self-hosted).

Decidimos restaurar **dentro da mesma conexão viva**, via `ATTACH DATABASE`: o arquivo `.sqlite` enviado é primeiro validado isoladamente (conexão própria `readonly`, `PRAGMA integrity_check` + confirma a tabela `registros`), depois anexado como schema `origem` na conexão principal (`db`), e o conteúdo de cada tabela do schema atual é trocado por `DELETE FROM main.tabela` + `INSERT INTO main.tabela SELECT * FROM origem.tabela`, numa única transação (`db.transaction`), com `foreign_keys` desligado durante a troca (evita depender de ordem entre DELETE/INSERT) e `PRAGMA foreign_key_check` rodando **dentro** dessa mesma transação — qualquer violação lança e o `better-sqlite3` desfaz tudo automaticamente, sem deixar o banco vivo num estado parcial. Como schema e conexão nunca mudam (só as linhas), os `Statement`s já preparados em todos os repositories continuam válidos sem qualquer reinício. Nomes de tabela pro `DELETE`/`INSERT` vêm sempre do schema **vivo** (`sqlite_master` da própria conexão, nunca do arquivo enviado) — evita injeção de SQL via nome de tabela malicioso num arquivo forjado.

Efeito colateral corrigido durante a implementação: `new Database(buffer, { readonly: true })` (validar direto de um `Buffer` em memória, sem tocar disco) falha com `SQLITE_CANTOPEN` na primeira query — `{ readonly: true }` não é suportado no modo de abertura via `Buffer` (`sqlite3_deserialize`) do `better-sqlite3`. A validação sempre grava um arquivo temporário (`os.tmpdir()`) primeiro, mesmo caminho reaproveitado pelo `ATTACH`. Outro efeito colateral, do lado do cliente: SvelteKit trata o corpo da requisição como ausente sem `content-length`/`transfer-encoding`, e `.sqlite` não é um MIME type reconhecido pela maioria dos navegadores — um `<input type=file>` normal manda `File.type = ""`, sem `Content-Type`, e o upload chegava vazio no servidor. O formulário de restauração seta `Content-Type: application/octet-stream` explicitamente.

Alternativas rejeitadas: **reiniciar o processo após a restauração** (`db.close()` + sobrescrever o arquivo + `process.exit()`) — mais simples de implementar, mas exige política de restart do container (`--restart unless-stopped`) que não é garantida em toda instalação self-hosted (o `docker run` documentado no README não a define por padrão), deixaria a aplicação fora do ar até reinício manual em ambientes sem essa política, e ainda corre risco de responder a requisição antes do processo morrer. **Refatorar todos os repositories para preparar `Statement`s por chamada** (em vez de uma vez no import), permitindo trocar a conexão livremente como no `notas` — rejeitada por ser uma reescrita invasiva de ~9 arquivos só pra suportar uma operação administrativa rara, desfazendo um padrão de performance deliberado (preparar uma vez, reusar) sem necessidade.

`BODY_SIZE_LIMIT` do adapter-node (padrão 512K) precisou subir — `Dockerfile` define `20M`, suficiente pro SQLite (só metadados, sem BLOBs no schema).
