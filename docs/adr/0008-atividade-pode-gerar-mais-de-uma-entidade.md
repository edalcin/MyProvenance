# Atividade pode gerar mais de uma Entidade

O modelo original (CONTEXT.md, especificação inicial) limitava Criação e Transformação a gerar exatamente 1 Entidade nova, e Análise a 0 ou 1. Na prática, processos reais de pesquisa frequentemente produzem múltiplos artefatos de uma vez: uma Criação pode capturar várias planilhas de campo no mesmo evento de coleta; uma Transformação pode dividir um dataset bruto em treino/teste ou gerar versões derivadas em paralelo (ex.: uma limpa e uma agregada); uma Análise pode produzir vários artefatos de saída (gráfico, tabela, relatório) na mesma execução. Forçar o usuário a registrar uma Atividade separada por Entidade gerada, todas com o mesmo Agente/data/processo, seria redundante e não refletiria o evento real. Decidimos que os três tipos de Atividade podem gerar 0 (só Análise), 1 ou mais Entidades — `entidadeGeradaId: UUIDv7 | null` virou `entidadesGeradas: UUIDv7[]`. Regra de cardinalidade revisada: Criação usa 0 gera 1+; Transformação usa 1+ gera 1+; Análise usa 1+ gera 0+. O diagrama Mermaid passa a desenhar uma seta de cada Entidade usada para cada Entidade gerada (fan-out completo) quando há múltiplas entradas e/ou saídas. `schemaVersion` do JSON exportado subiu para 2; uploads com `schemaVersion` 1 continuam aceitos (ADR-0005) — o campo antigo `entidadeGeradaId` é ignorado na leitura, a proveniência real de cada Entidade vem do seu próprio `geradaPorAtividadeId`, nunca duplicada na Atividade. Alternativa rejeitada: manter cardinalidade 1 e exigir uma Atividade por Entidade gerada — rejeitada por multiplicar registros redundantes (mesmo Agente/data/processo repetidos) sem ganho de fidelidade histórica.
